# CSP `unsafe-eval` Audit — 2026-05-04

**Ticket**: Phase 5 Slice 7 (SEC-12)
**Auditor**: Implementation automated + manual static analysis
**Outcome**: `unsafe-eval` **removed** from `internal/api/rest/router.go`

---

## Background

The API server's `securityHeaders` middleware set:

```
Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-eval'; ...
```

`unsafe-eval` permits `eval()`, `new Function()`, and `setTimeout(string)` in
browser-side JavaScript. It is the broadest relaxation of script execution and
should only be present if something in production genuinely requires it.

---

## Audit scope

The CSP header is applied by `securityHeaders` in `internal/api/rest/router.go`
(line ~1447) to **every response from the Go API server**. The API serves JSON
and GraphQL — not HTML pages. The Next.js web frontend runs as a separate
container and manages its own response headers.

Practically: `script-src` only matters for responses that contain inline
`<script>` tags or are loaded as scripts. The API delivers JSON, so the
`unsafe-eval` directive in the API's CSP is a no-op for API consumers.
It is removed anyway to align the stated policy with the actual need.

---

## Static analysis

**Tool**: `grep` over `web/src/` (TypeScript / TSX source)

| Pattern | Files matched | Notes |
|---|---|---|
| `eval(` | 0 | No direct eval calls |
| `new Function` | 0 | No dynamic code generation |
| `mermaid` | 2 | `ArchitectureDiagram.tsx`, `diagram-utils.ts` |

**Mermaid**: version `^11.13.0` (locked in `web/package.json`). Audited
`web/node_modules/mermaid/dist/mermaid.core.mjs` and all 199 chunks under
`web/node_modules/mermaid/dist/chunks/`. Zero occurrences of `eval(` or
`new Function` in any chunk. Mermaid 11.x dropped the `securityLevel: "antiscript"`
mode that used `new Function` for sanitization; the current renderer uses
DOMPurify instead.

**PostHog** (`posthog-js`): audited `node_modules/posthog-js/dist/`. Zero
eval occurrences.

**Next.js**: Next.js 15's production bundle (`next build`) does not use
`eval`. Dev-mode webpack hot-reload uses eval for sourcemaps, but that
runs in the web container under Next.js's own CSP management, not under
the Go API's `securityHeaders` middleware.

---

## Manual verification

The CSP header in `securityHeaders` applies to API server responses
(`/api/*`, `/auth/*`, `/graphql`, `/healthz`, etc.). All of these return
JSON or plain text — never inline scripts. A browser loading one of these
endpoints would apply `script-src` to any `<script>` tags in the response
body; there are none.

The web frontend (`web/` container, port 3000) is served by the Next.js
standalone server. Its responses are not subject to the Go API's CSP
header. Next.js production mode has no `eval` requirement; if it did,
fixing it would require a CSP header in the web container's Next.js
config (`next.config.ts`), not in the API router.

---

## Decision

**Remove `'unsafe-eval'`** from `script-src` in `internal/api/rest/router.go`.

The updated directive:

```
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; connect-src 'self'; form-action 'self' https:
```

`'unsafe-inline'` in `style-src` is retained — that was never the audit
target and removing it would require adding CSP nonces to all inline styles
(a larger refactor, out of scope for this slice).

---

## Future work

- Consider adding CSP nonces for inline styles to eventually remove
  `'unsafe-inline'` from `style-src`.

## Follow-up: CI enforcement

**Tracking ticket**: [CA-156](https://plane.xmojo.net/agile-solutions-group/projects/d3fa4bd8-1177-4364-88a7-aae69698b75d/issues/f71f2a1d-cee6-4551-9dd4-b3cf0b58f79f/)

The removal of `unsafe-eval` is not enforced by CI. A future dependency
update could reintroduce an eval-requiring code path without detection.

CA-156 tracks adding an automated check — either a headless-browser CSP
violation scan (Next.js production build + Playwright) or a post-build
static scan of the compiled bundle for `eval(` and `new Function(` — that
fails the CI pipeline before such a regression reaches `main`.
